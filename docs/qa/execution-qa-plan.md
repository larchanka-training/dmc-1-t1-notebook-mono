# Execution Test Strategy (Runtime QA Plan)

## Назначение

Документ описывает стратегию тестирования **runtime-слоя** notebook — той части системы, которая получает исходный код ячейки от пользователя и возвращает результат выполнения (output, ошибку, поток stdout/stderr, статус ячейки).

Цель — гарантировать, что выполнение пользовательского кода в notebook:

- даёт **предсказуемый и корректный** результат для валидного кода;
- **корректно сообщает об ошибках** (синтаксические и runtime) без падения UI;
- **не деградирует по производительности** при типовых нагрузках;
- **не ломает уже работающие сценарии** при изменениях в kernel/UI.

Документ дополняет общий `qa-plan.md` и используется при ревью PR, затрагивающих runtime, исполнение ячеек или их output.

## Область покрытия (Scope)

В скоупе:

- Модуль исполнения кода ячейки (`ui/src/features/notebook/lib/fakeKernel.ts` сейчас, реальный kernel — в будущем).
- Хук запуска ячейки `useRunCell` и связанные actions/reducer (`notebookContext`, `notebookSlice`).
- Отображение результата в UI ячейки (output: `stream`, `execute_result`, `error`).
- Состояния выполнения ячейки: `idle` / `queued` / `running` / `error`.
- Поле `executionCount` и его монотонность.

Вне скоупа:

- Markdown-/Raw-ячейки (не исполняются).
- Загрузка/сохранение notebook на бэкенде (`api/`) — покрывается отдельным API-планом.
- Аутентификация, роутинг, общая вёрстка приложения.

## Предусловия и тестовые окружения

- **dev (локально)** — основное место прогона ручных и автоматических проверок.
- **prod** — только smoke по критическим сценариям после релиза.
- Браузеры для ручной проверки: последняя стабильная Chrome (обязательно), Firefox (выборочно).
- Тестовые данные — синтетический JS-код, оформленный как набор сниппетов (см. ниже).

## Принципы тестирования runtime

- **Контракт важнее реализации.** Проверяем форму выхода (`CellOutput`, `status`, `executionCount`), а не конкретные внутренности kernel.
- **Изолируем ячейку.** Ошибка в одной ячейке не должна ронять весь notebook или блокировать запуск других ячеек.
- **UI всегда отражает состояние.** Любой запуск меняет `executionState` через `running` и завершается либо `idle`+output, либо `error`+output типа `error`.
- **Никаких «тихих» падений.** Любой сбой исполнения — это видимый пользователю output, а не исключение в консоли.

## Тестовые сценарии — синтаксические ошибки

Цель: код не запускается, пользователь видит понятную ошибку, UI остаётся работоспособным.

| ID | Сценарий | Вход | Ожидаемое поведение |
|---|---|---|---|
| EXE-SYN-01 | Незакрытая скобка | `console.log("x"` | output типа `error`, `ename` указывает на синтаксис (`SyntaxError`), `status=error`, ячейка в состоянии `error` |
| EXE-SYN-02 | Незакрытая строка | `const a = "abc` | как EXE-SYN-01 |
| EXE-SYN-03 | Зарезервированное слово как имя | `const class = 1` | как EXE-SYN-01 |
| EXE-SYN-04 | Неверный токен | `const a = ;` | как EXE-SYN-01 |
| EXE-SYN-05 | Битый шаблонный литерал | `` `unterminated `` | как EXE-SYN-01 |
| EXE-SYN-06 | После ошибки запуск другой валидной ячейки | EXE-SYN-01 → EXE-OK-01 | вторая ячейка выполняется успешно, `executionCount` продолжает расти |

> Замечание для текущей заглушки `fakeKernel.ts`: она не различает синтаксические ошибки. Этот блок становится обязательным с момента подключения реального движка. До этого сценарии EXE-SYN-* проверяются как **известное ограничение** и фиксируются в issue tracker.

## Производительность (Performance sanity)

Цель — не точные бенчмарки, а **санитарные пороги**, при превышении которых заводится баг.

| ID | Сценарий | Условие | Порог (ориентир) |
|---|---|---|---|
| EXE-PERF-01 | Запуск простой ячейки (`1+1`) | холодный запуск | UI показывает результат < **1.5 c** |
| EXE-PERF-02 | Запуск простой ячейки повторно | прогретый kernel | < **800 мс** |
| EXE-PERF-03 | Последовательный запуск 10 ячеек | в один клик / по очереди | все завершаются, порядок `executionCount` сохранён, общее время < **10 c** |
| EXE-PERF-04 | Большой output (`console.log` в цикле, ~1000 строк) | один запуск | UI не зависает > **2 c**, output отображается полностью или с явной индикацией усечения |
| EXE-PERF-05 | Длинная строка в `execute_result` (~100 KB) | один запуск | без падения вкладки, без зависания > **2 c** |
| EXE-PERF-06 | Долгий код (`while(Date.now() - t < 3000){}`) | один запуск | ячейка в состоянии `running` всё это время, UI остаётся отзывчивым (скролл, переключение ячеек) |
| EXE-PERF-07 | Подряд 50 быстрых запусков | стресс | нет утечек состояния: `executionCount` корректен, кнопка Run снова активна |

Метрики снимаются вручную (секундомер / DevTools Performance) до появления автоматических замеров.

## Контракт output ячейки (что проверять всегда)

При любом запуске:

- Объект результата соответствует `CellOutput`: один из `stream | execute_result | error`.
- Для `error` все три поля заполнены: `ename`, `evalue`, `traceback` (массив строк, не пустой).
- Для `stream` поле `stream` имеет значение `stdout` или `stderr`, `text` — строка.
- `executionCount` — число, **строго больше** предыдущего значения той же сессии.
- `executionState` ячейки проходит через `running` и завершается:
  - `idle` — при `status=ok`;
  - `error` — при `status=error`.

## Регрессионный чек-лист runtime

Прогоняется при каждом PR, который трогает: `ui/src/features/notebook/lib/*`, `model/useRunCell.ts`, `model/notebookContext.tsx`, `model/notebookSlice.ts`, компоненты вывода ячейки.

Smoke (обязательный минимум, ~5 минут):

- [ ] Создаётся новая code-ячейка, в неё можно ввести код.
- [ ] Запуск ячейки с `1 + 1` показывает результат, `executionCount` = 1 (для свежей сессии).
- [ ] Запуск ячейки с `console.log("hi")` показывает `hi` в stdout.
- [ ] Запуск ячейки с заведомой ошибкой показывает блок ошибки с `ename`/`evalue`/traceback.
- [ ] После ошибки можно успешно запустить другую ячейку.
- [ ] Повторный запуск той же ячейки увеличивает `executionCount`.
- [ ] Во время выполнения ячейка визуально в состоянии `running`, по завершении — `idle`/`error`.

Расширенный регресс (при изменениях в kernel / output rendering):

- [ ] Все сценарии EXE-OK-01..10.
- [ ] Все сценарии EXE-RT-01..07.
- [ ] EXE-SYN-* (если реальный kernel подключён).
- [ ] EXE-PERF-01..03 укладываются в пороги.
- [ ] Output типа `stream` и `execute_result` не подменяют друг друга при последовательных запусках.
- [ ] Большой output (EXE-PERF-04) не ломает скролл и не зависает UI.
- [ ] Параллельных запусков одной ячейки не происходит (повторный клик Run во время `running` либо игнорируется, либо ставится в очередь — поведение зафиксировано).
- [ ] Нет ошибок в DevTools Console при штатных и ошибочных запусках.

## Текущие известные ограничения

- `fakeKernel.ts` не исполняет реальный JS: распознаёт лишь подстроку `error` и шаблон `console.log("...")`. Сценарии EXE-SYN-* и большинство EXE-RT-* в текущей реализации **детерминированно не воспроизводятся** — план применяется в полном объёме при подключении реального движка. До этого они служат списком требований к будущему kernel.
- `executionCount` сейчас глобальный счётчик в модуле и не сбрасывается между «сессиями» — учитывать при проверке EXE-OK-07/08.
