# Архитектура исполнения кода (Execution Architecture)

## 1. Обзор
Этот документ определяет модель исполнения кода для приложения JavaScript Notebook. Он описывает, как JavaScript-код, написанный пользователем в ячейках блокнота, выполняется безопасно, эффективно и с правильным управлением состоянием.

## 2. Решение: Комбинированный подход Web Worker + QuickJS WASM

После оценки нескольких альтернатив было принято решение использовать **комбинированный подход**:
Код пользователя будет вычисляться с помощью **WebAssembly (WASM) сборки QuickJS**, запущенной внутри **Web Worker**.

### 2.1 Рассмотренные альтернативы

| Альтернатива | Описание | Почему отклонена |
|---|---|---|
| **iframe Sandbox** | Исполнение кода через `eval()` в скрытом `<iframe sandbox="allow-scripts">`. | **Слабая изоляция:** Сохраняется доступ к некоторым браузерным API. **Потеря состояния:** Если скрипт зависает, удаление iframe для его остановки уничтожает все переменные и состояние. |
| **Raw Web Worker** | Запуск кода напрямую в потоке Web Worker. | **Безопасность:** Web Worker имеют доступ к `fetch`, `WebSockets` и `IndexedDB`, что затрудняет предотвращение кражи данных. **Потеря состояния:** Остановка бесконечного цикла требует `worker.terminate()`, что полностью очищает память. |
| **Server-Side (Lambda/Docker)** | Отправка кода на бэкенд для исполнения. | **Latency:** Высокое время отклика для простых операций. **Сложность/Стоимость:** Требует мощной инфраструктуры для изоляции (Firecracker/gVisor) и сложной синхронизации состояния между клиентом и сервером. |

### 2.2 Выбранное решение: QuickJS WASM + Web Worker

#### WebAssembly (WASM) Runtime: `quickjs-emscripten`
*   **Движок:** [QuickJS](https://bellard.org/quickjs/), скомпилированный в WebAssembly через Emscripten (пакет `justjake/quickjs-emscripten`).
*   **Почему:** QuickJS — это легкий и полностью совместимый со стандартами движок JS. Компиляция в WASM обеспечивает максимальную изоляцию ("песочницу"). По умолчанию среда имеет нулевой доступ к API браузера (нет `window`, `document`, `fetch` и т.д.). Мы явно пробрасываем только то, что безопасно и необходимо (например, кастомную реализацию `console.log`).

#### Песочница и многопоточность: Web Worker
*   **Среда:** Контейнер WASM будет инициализирован внутри выделенного потока Web Worker.
*   **Почему:** Хотя WASM обеспечивает изоляцию кода, выполнение тяжелых вычислений или бесконечных циклов в основном потоке заморозило бы React UI. Обернув инстанс WASM в Web Worker, мы достигаем истинной асинхронности исполнения без блокировки пользовательского интерфейса.

### 2.3 Плюсы и минусы

**Плюсы:**
*   **Безопасность (Zero-Trust):** Полная изоляция от DOM браузера и чувствительных API.
*   **Неблокирующий UI:** Тяжелые вычисления не вызывают лагов интерфейса блокнота.
*   **Безопасные прерывания:** Мы можем останавливать бесконечные циклы *без* уничтожения среды и без потери объявленных переменных.
*   **Стабильная среда:** Поведение JS идентично во всех браузерах, независимо от их нативной версии V8 или SpiderMonkey.

**Минусы:**
*   **Оверхед производительности:** Интерпретация JS внутри JS-движка на базе WASM медленнее, чем нативное исполнение в V8 (однако это незаметно для большинства задач блокнота).
*   **Размер бандла:** Добавляет около 1 МБ к фронтенд-бандлу из-за бинарного файла WASM.
*   **Отставание по фичам:** QuickJS поддерживает ES2023, но может отставать от самых новых экспериментальных фич JS по сравнению с современными браузерами.

## 3. Управление состоянием и персистентность
*   **Состояние в памяти:** Модуль QuickJS WASM сохраняет свою внутреннюю кучу памяти (heap) между выполнениями ячеек. Переменные, объявленные в "Ячейке 1", будут доступны при вычислении "Ячейки 2", пока страница не будет обновлена.
*   **Персистентность между сессиями:** Для MVP создание снимков памяти (snapshotting — сохранение точного состояния переменных в БД) находится **вне рамок (out of scope)**. Если пользователь обновит страницу или вернется на следующий день, он должен использовать функцию "Run All" для последовательного перезапуска ячеек и восстановления состояния.
*   *Примечание: Изучалась возможность нативного WASM snapshotting (`quickjs-wasi`), но необходимость в тяжелых браузерных полифилах для WASI делает это решение неподходящим для MVP.*

## 4. Модель коммуникации

Взаимодействие между UI (Main Thread) и средой исполнения (Web Worker) использует стандартный браузерный API `postMessage`.

*   **UI -> Worker:** Отправляет запросы на выполнение, содержащие ID ячейки и строку исходного кода.
*   **Worker -> UI:** Стримит вывод консоли, возвращает финальные результаты вычислений или сообщает об ошибках.

*Перспектива:* Если протокол сообщений станет сложным, мы можем внедрить RPC-обертку, такую как `Comlink`, поверх Web Worker.

## 5. Стратегия обработки ошибок

1.  **Синтаксические ошибки:** Отлавливаются QuickJS на этапе парсинга. Исполнение немедленно останавливается, и ошибка с номерами строк возвращается в UI.
2.  **Ошибки рантайма:** Исключения (например, `TypeError`, `ReferenceError`) перехватываются внутри контейнера WASM. Сообщение об ошибке и stack trace сериализуются и отправляются через `postMessage`.
3.  **Бесконечные циклы (Timeouts):**
    *   В отличие от обычных Web Worker, где `terminate()` уничтожает все состояние, QuickJS поддерживает **обработчик прерываний (interrupt handler)**.
    *   Мы настроим порог таймаута (например, 5 секунд) на каждую ячейку.
    *   Если код превышает это время, QuickJS прервет вычисление, выбросив ошибку. Критическое преимущество здесь в том, что **состояние памяти WASM (переменные из предыдущих ячеек) остается нетронутым.**

## 6. Диаграмма потока исполнения (Execution Flow)

```mermaid
sequenceDiagram
    participant U as UI (React)
    participant W as Web Worker
    participant Q as QuickJS (WASM)

    Note over U,Q: Инициализация (Загрузка страницы)
    U->>W: Spawn Worker
    W->>Q: Load WASM & Instantiate VM
    
    Note over U,Q: Выполнение ячейки
    U->>W: postMessage({ type: 'EXECUTE', cellId: '1', code: 'let x = 10;' })
    
    W->>Q: Start Timer (5s)
    W->>Q: vm.evalCode('let x = 10;')
    
    alt Успех
        Q-->>W: Result (undefined)
        W-->>U: postMessage({ type: 'SUCCESS', cellId: '1', result: 'undefined' })
    else Ошибка Syntax/Runtime
        Q-->>W: Throw Error
        W-->>U: postMessage({ type: 'ERROR', cellId: '1', error: 'SyntaxError...' })
    else Таймаут (Бесконечный цикл)
        Q-->>W: Interrupt Triggered
        W-->>U: postMessage({ type: 'TIMEOUT', cellId: '1' })
    end
```
