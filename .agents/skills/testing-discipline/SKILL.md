# Skill: testing-discipline

Правила тестирования в проекте.

## Когда использовать

- Пишешь или меняешь код в `api/` или `ui/` — нужен тест на новое поведение.
- Готовишь задачу к коммиту — нужны зелёные тесты.
- Чинишь баг — нужен падающий тест, воспроизводящий дефект.

**Не использовать** для чисто-документационных задач.

## Алгоритм

### API (pytest)

1. Размести тест зеркально коду: `tests/test_<resource>.py`.
2. Используй `pytest` с моками через `unittest.mock` (`AsyncMock`, `MagicMock`).
3. Замокай все внешние системы (никакой реальной сети):
   - БД: `mocker.patch` или mock `get_db` dependency;
   - AWS Bedrock: `mocker.patch("app.ai.bedrock.invoke_model")`;
   - JWT: mock `get_current_user`.
4. Покрой happy path, ошибки домена (HTTPException с правильным статусом), границы (пустой результат, превышение лимита).
5. Прогон: `ruff check . && pytest` — зелёный.

### UI (Vitest)

1. Размести тест рядом с файлом: `<file>.test.ts` / `<file>.test.tsx`.
2. Используй `vitest` + `@testing-library/react` для компонентов.
3. Замокай API вызовы через `vi.mock("../api/service")`.
4. Покрой happy path, edge cases (пустые данные, ошибки загрузки).
5. Прогон: `npm run lint && npm run test && npm run build` — зелёный.

### E2E (Playwright)

- Для UI-флоу (взаимодействие пользователя) — добавить Playwright E2E тест.
- Запускать через Docker Compose (полный стек).

## Чего избегать

- Реальных вызовов API / БД / интернета в unit-тестах.
- Маскировки падений через `skip` вместо починки.
- Ослабления или удаления существующих тестов без явного указания.
- Недетерминизма (зависимость от времени, порядка, внешних данных).
