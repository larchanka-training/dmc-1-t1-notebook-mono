# Skill: documentation-discipline

Правила работы с документацией проекта.

## Когда использовать

- Меняешь поведение системы — нужно обновить документацию до или вместе с кодовым коммитом.
- Правишь `docs/`, `README.md`, `AGENTS.md`, `.agents/`.
- Переименовываешь или переносишь файл — нужно поправить ссылки.

**Не использовать** для правок кода без изменения поведения.

## Алгоритм

1. **Только относительные пути.** Запрещены абсолютные пути ФС (`/home/...`, `/Users/...`, `C:\...`). Все ссылки — от корня проекта (`docs/...`, `api/...`, `./architecture.md`).
2. **Язык — русский** по умолчанию; технические идентификаторы (имена модулей, функций, env, путей) — латиницей.
3. **Doc-before-code.** Поведение сначала описывается в профильном документе (`docs/architecture/`), затем меняется код. Кодовый коммит, меняющий поведение, без обновления документации не принимается.
4. **Синхронизация ссылок.** При переименовании/удалении файла — `grep -rn` по репозиторию и починить все живые ссылки.
5. **Источник истины — код.** При расхождении документ приводится к коду отдельным `docs(...)`-коммитом.
6. **AGENTS.md — единая точка входа** для всех AI-агентов. Symlink'и (`CLAUDE.md`, `GEMINI.md`, `QWEN.md`, `.github/copilot-instructions.md`) ссылаются на него. Правки правил — только в `AGENTS.md`.

## Чего избегать

- Абсолютных путей ФС в любой документации.
- «Висячих» ссылок после переименования — всегда проверяй `grep -rn`.
- Кодового коммита, меняющего поведение, без обновления документации.
- Дублирования правил в нескольких файлах — держи один источник истины.
