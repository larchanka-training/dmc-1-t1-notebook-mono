# Skill: code-review-discipline

Правила ревью кода для PR.

## Когда использовать

- Ревью PR (свой или чужой).
- Самопроверка перед коммитом.

## Чек-лист

1. **Хирургичность.** Каждая изменённая строка напрямую относится к задаче. Нет «улучшений» соседнего кода.
2. **Тесты.** Новое поведение покрыто тестами. Багфикс включает воспроизводящий тест. Тесты зелёные.
3. **Стиль.** API: `ruff check .` — 0 ошибок. UI: `npm run lint` — 0 предупреждений.
4. **Архитектура.** Слои не нарушены (UI→API→Proxy). Нет бизнес-логики в Proxy. Нет обращения UI к БД в обход API.
5. **Ошибки.** Ожидаемые исключения ловятся. Пользователю — человеческое сообщение. Stacktrace — только в лог.
6. **Безопасность.** Нет секретов в коде. Prompt injection проверяется. JWT валидируется. Rate limits работают.
7. **Документация.** Поведение обновлено в `docs/`. Ссылки не висячие. Относительные пути.
8. **Submodule.** Pointer обновлён в mono. Commit внутри submodule существует.
9. **Коммит.** Атомарные коммиты. Формат проекта. Тема ≤ 50 символов.

## Чего избегать

- Апрува без проверки тестов.
- Игнорирования красного CI.
- Слияния без обновления документации при изменении поведения.
