# Skill: error-handling-discipline

Правила обработки ошибок в API и UI.

## Когда использовать

- Пишешь или меняешь endpoint, сервис, React-компонент с внешним I/O.
- Добавляешь новый класс ошибок или новую внешнюю интеграцию.
- Формируешь сообщение пользователю при сбое.

## Алгоритм

### API (FastAPI)

1. **Нет необработанных исключений.** Каждый endpoint ловит ожидаемые ошибки и превращает в `HTTPException` с правильным статусом.
2. **Статус-коды:**
   - 400 — невалидный ввод (не путать с 422 Pydantic);
   - 401 — не авторизован;
   - 403 — нет прав;
   - 404 — не найдено;
   - 422 — ошибка валидации / невалидный результат AI;
   - 429 — rate limit;
   - 503 — внешняя зависимость недоступна (Bedrock).
3. **Логирование.** `logger.exception(...)` или `logger.error(..., exc_info=True)` — stacktrace только в лог. Чувствительные данные в лог не пишем.
4. **Деградация.** Где возможно — WARNING и продолжение, а не падение.

### UI (React)

1. **Error boundaries** на уровне страницы — пользователь видит понятное сообщение, не белый экран.
2. **API ошибки.** `apiClient` централизованно обрабатывает 401 (refresh), 429 (сообщение), 5xx (сообщение).
3. **Пользователю — человеческое сообщение**, без stacktrace и внутренних путей.
4. **Тихая обработка для analytics.** `useAnalytics` не должен ронять UI при ошибке отправки события.

## Чего избегать

- Голого `except Exception: pass` и проглоченных ошибок без лога.
- Stacktrace в сообщении пользователю.
- Возврата 500 на ожидаемые ошибки домена.
- Падения UI из-за ошибки аналитики.
